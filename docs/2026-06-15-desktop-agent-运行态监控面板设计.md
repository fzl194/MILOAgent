## ① 功能要点

**解决什么问题:** 现有 trace(事后时间线)和统计面板看不到「每次请求实际发给模型的那串完整消息」和「上下文度量(token 估算、窗口填充率、触发了哪些压缩、裁掉了什么)」——而这正是判断上下文工程对不对最需要看的。现有 trace 只存增量、是事后快照、不可实时观察。

**面板做什么:**

- **三大监控维度**:① 上下文(每次请求的实际发送视图 + 裁剪决策);② token(估算量、窗口填充率、真实用量);③ 工具调用(参数→结果→风险→审批→耗时的全链,可关联回触发它的那次请求)。
- **实时 + 可追溯**:开发时边跑边看;事后也能回放任意会话的任意一次请求,看模型当时看到的那串。
- **形态:主应用内面板**(admin 区延伸,与现有 TraceTimeline / StatsPanel 同处一个面板区)。**不另开独立窗口**——独立窗口带来的 main 进程窗口管理 + 全新一套跨窗口推送 IPC,对个人项目过重,且采集层已做到「永不抛」,崩溃隔离价值有限。独立窗口降级为改进方向。
- **可热插拔**:面板上勾选开关某类监控;加新监控维度只「新增」不改 agent 核心。
- **低侵入 + 演进友好**:默认关闭时 agent 行为零开销;采集固化在循环 / 策略**主动暴露的语义方法**上,而非读局部变量,降低与内部实现的耦合。

## ② 实现逻辑

整体分三层:**采集接口(进程级单例总线)→ 内存发布-订阅 → 消费层(实时面板 / 持久化)**。砍掉独立窗口后,全程在同一 renderer 进程内,不再需要任何跨窗口推送 IPC。

### 采集接口 = 进程级单例总线

- **进程级单例**:总线在模块加载时就绪,生命周期 = renderer 进程;不随 turn 重建(AgentLoop 虽然每个 turn 新建,但直接 import 这个单例总线,订阅跨 turn 存活)。
- **惰性发布**:发布方法接收一个「payload 工厂」而非值——当某维度**既无订阅者、又不需要持久化**时,工厂根本不被调用,最贵的「完整请求视图」在没人看时零成本。
- **内部 fan-out**:发布一次,多个订阅者(实时面板、持久化、未来的导出等)各收一份,采集点不会被重复触发。
- **总线统一兜异常**:发布 / 分发逻辑在总线层 try/catch,采集实现或订阅者抛出的任何错误都被吞掉并告警——「永不抛」是总线契约,采集点调用处无需再包 try。

### 采集点:固化在「主动暴露的语义方法」上(审查关键修订)

不读 agent 内部的局部变量、不重复调用带副作用的派生方法,而是让循环 / 策略**主动把数据交出来**:

- **请求视图**:agent 循环在组装完请求、即将发出时,通过一个注入的回调(语义为「请求已构造好」)把「实际要发给模型的视图 + 这次请求的标识」主动交给采集层。循环内部怎么重构都不影响——只要它还发请求,就会触发这个回调。
- **上下文度量 + 裁剪决策**:在上下文策略上**新增一个可选方法**(语义为「产出视图并附带裁剪决策」),返回视图 + 每个压缩步骤的决策(谁触发了、折叠 / 裁掉了哪些块)。该方法默认退化为「只返回视图」(行为等同今天),采集走它才能拿到真实的裁剪决策。这正是「接口只加方法、不改老方法」演进原则的落地处。
- **工具执行**:复用现有的工具执行产出点(已有 tool 执行事件),不新增采集位。
- **abort / 停止**:在用户中止、循环提前 return 的路径上补发一个「turn 已中止」事件,避免面板上出现「请求已发出但永远等不到完成」的悬挂 turn。
- **稳定信封**:每条事件固定携带 `{turnId, callId, round, sessionId, ts, dimension}`,业务细节放宽松的 `details`——并发 turn、模型中途切换时也能正确归档,不串台。

### 维度注册表 + 加维度三步

所有维度集中声明(名字、标签、分组、持久化策略),面板据此**动态枚举**开关;agent 未来新增维度,面板自动出现。**加一个新维度 = 三步全新增**:① 注册表声明 → ② 对应产出点加一行采集 → ③ 面板加一个渲染组件。不改接口、不改总线、不改老代码。

初始五个维度:请求视图、上下文度量(含裁剪决策)、工具调用、token 用量、turn 生命周期。其中 **token 用量、工具调用是现有 trace 的「只读视图」,零额外存储**;只有**请求视图 + 上下文度量(含裁剪决策)**是真新增数据,落在请求快照里。

### 进程模型(砍窗口后的简化)

- 全程同一 renderer 进程:总线本地 fan-out → 面板组件(经状态库订阅)实时显示;持久化订阅者同进程写盘。
- **不再需要跨窗口推送 IPC、不需要 main 进程做订阅中转**——审查指出的「现有 IPC 全是请求-响应、没有推送模式」这一缺口随之消失。
- 面板是纯订阅者,且采集永不抛:面板自身的异常不会反向影响 agent。

### 持久化(按维度路由 + 时序 + 容错)

- **请求视图 + 上下文度量**:合并存成一份「请求快照」文件(每次请求一份,含完整视图 + 裁剪决策 + 度量),事后可整笔回放「模型当时看到的那串 + 为什么」。
- **token 用量 / 工具调用**:直接复用现有 trace 与统计,不重复存储。
- **turn 生命周期**:轻量追加到现有 trace。
- **时序与容错**:持久化订阅者**异步串行**消费,先写快照、成功后才写 trace 指针;两步之间无原子性,中途崩溃可能留孤儿快照或缺失快照——回看侧**容忍快照缺失**(指针指向的文件不存在时降级为「只显示度量,视图不可用」,不报错);快照写失败在面板上**可见标记**,不静默吞掉。
- **超大视图**:持久化与面板渲染侧都设阈值,超限截断并标记。

### 监控面板(主应用内)

- **位置**:admin 面板区新增「监控」视图,与 TraceTimeline / StatsPanel 并列(或可折叠侧栏)。
- **顶栏**:会话选择、`实时 ↔ 回看` 切换、维度开关(动态来自注册表)。
- **左侧事件流**:实时模式滚动推送;回看模式从磁盘加载该会话的请求快照列表,点开某次。
- **右侧详情 / 「上下文工程」视图(核心)**:选某次请求,把完整视图渲染成可读对话序列(角色 + 内容 + 折叠 / 被裁标记),旁列度量仪表盘(token 估算对窗口的填充条、触发了哪些压缩、被折叠 / 裁掉的块)。一眼验证「模型当时看到的是不是符合我的意图」。

### 两条贯穿原则(诚实化)

1. **低侵入(非零侵入)**:默认关闭零开销 + 惰性 payload + 总线吞异常 + 面板独立订阅。**诚实承认接入时需改 4 个现有文件**(agent 主循环加请求回调、上下文策略加「视图+决策」方法、共享类型加监控事件 / 快照指针、trace 埋点处接快照指针),外加全新模块(总线、面板、持久化订阅者)。措辞用「低侵入 + 默认关闭时零开销」,避免「零侵入」误导后续维护者低估 review 责任。
2. **演进友好**:依赖抽象不依赖具体;采集固化在「循环 / 策略主动暴露的语义方法」上而非局部变量;事件「稳定信封 + 宽 payload」;接口只加方法不改老方法;渲染数据驱动。**诚实说明**:请求视图与上下文度量这两个最核心维度,贴在「组装请求」「产出视图」这两个概念上,而它们正是 agent 未来 P2(handoff 摘要)/ P3(cache-aware 裁剪)重构时最可能动的位置——通过「主动暴露的语义方法」把耦合降到语义层,使演进成本限于「挪一个回调 / 改一个方法的内部实现」,而非重写监控。

## ③ 当前弊端

- **token 仍是粗估**:继承现有启发式估算,面板显示的是估算值而非精确 token(多后端分词差异本身存在)。
- **单个超大消息仍可能撑满窗口**:总量兜底只保总预算,一条巨型工具结果仍会被发出;监控能看到问题,治本要在源头限制工具结果大小(另一项独立工作)。
- **请求快照会累积 + 持久化非强一致**:高频长会话积累快照文件需清理策略;两步写(快照 → 指针)无原子性,靠回看侧容错降级,不是强一致。
- **核心采集点贴在相对易动的概念上**:请求视图 / 上下文度量落在「组装请求 / 产出视图」概念,是 P2 / P3 重构的高概率改动区;已用「主动暴露语义方法」降低耦合,但不是绝对零演进成本。
- **主应用内面板同进程**:面板渲染若卡顿理论上可能影响主 UI 响应;但采集永不抛 + 面板轻量,实测预期可控,真出现问题再拆独立窗口。
- **暂未覆盖 P2 / P3 压缩维度**:handoff 摘要、cache-aware 裁剪等尚未落地,对应监控维度等机制实现后再按「三步」补。

## ④ 改进方向

- 接入真实 tokenizer 替换粗估,token 面板转精确值。
- 源头限制工具结果大小,治本「超大消息撑满窗口」。
- 请求快照的保留 / 清理策略(按会话、时间或数量);若需强一致再引入原子写(如临时文件 + rename)。
- 预留 P2 / P3 压缩维度的接入位(handoff 摘要、cache-aware 裁剪的监控)。
- 面板演进为可回放 / 对比(两次请求视图的 diff),进一步辅助上下文工程调试。
- **若实测发现面板拖累主 UI,再拆独立窗口**(届时才引入跨窗口推送 IPC 与窗口生命周期管理)。
- 采集总线长期沉淀为「通用探针」能力:导出、评测、回放等外部能力复用同一套发布-订阅。
