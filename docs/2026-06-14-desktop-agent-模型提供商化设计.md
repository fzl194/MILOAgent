# desktop-agent · 模型「提供商」化设计

> 日期:2026-06-14
> 性质:设计/决策记录。把现有扁平的「模型配置」升级为以「提供商」为主实体的模型,含模型列表自动获取、默认模型、前端单分组下拉切换。
> 关联:`docs/2026-06-14-desktop-agent-Anthropic格式兼容调研.md`(协议口子对接)、`docs/2026-06-13-desktop-agent-功能盘点.md`(「不感知模型能力差异」一条)。

---

## 一、功能要点(要做成什么样)

- **主实体从「模型」升级为「提供商」**:一个提供商 = 一组连接信息(name + baseUrl + key + 协议)+ 一串可用模型。
- **协议**:当前都按 OpenAI 兼容(`/chat/completions` + `/models`);`protocol` 字段留口子,未来加 Anthropic(对接调研文档)。
- **模型列表自动获取**:用配置的 key 调 `GET {baseUrl}/models` 拉真实模型 id;**同时支持手动加/改**(有的提供商不开放该接口或列表不全)。
- **每个提供商配一个「默认模型」**;全局再有一个「默认提供商」。新会话 → 默认提供商 + 其默认模型。
- **前端切换(已定:单分组下拉)**:顶栏一个「提供商 · 模型」芯片;点开按提供商分组 —— 点组标题=切该提供商(用其默认模型),点组内项=该提供商 + 具体模型。

> 已实测三个提供商均支持 `GET /models`(Bearer 鉴权):DeepSeek(2)、GLM(7)、Qwen/DashScope(215)。

---

## 二、实现逻辑(机制)

### 数据模型(功能层)
- **提供商**:name、baseUrl、apiKey、protocol(openai | anthropic[预留])、models[]、defaultModel、contextWindow 策略。
- **模型项**:id + contextWindow(来自内置对照表或手填;按「模型」而非「提供商」)。
- **会话**:记录 `providerId + modelId`(替代现在记单个模型配置 id)。
- **全局**:一个默认提供商。

### 模型列表获取
- 触发:添加/编辑提供商时 + 列表内「刷新」按钮。
- 动作:`GET {baseUrl}/models` → 取真实 id → 缓存进配置文件(每轮对话不重拉)。
- 手动层:可在自动列表之上增删改(覆盖)。
- 失败降级:网络/接口失败 → 用缓存或手动列表,不阻塞对话。

### 上下文窗口(contextWindow)
- `/models` 只返回 id,不返回窗口 → 用**内置常见模型对照表**(glm / deepseek / qwen / claude …)+ **手动覆盖**;缺省回落保守值。
- 窗口归属在「模型」粒度(glm-4.5 ≠ glm-5.1),喂给已有的 token 预算上下文管理。

### 前端交互(已选定:单分组下拉)
- 顶栏芯片显示「提供商 · 模型」(如 `GLM · glm-5.1`),当前选中带 ✓。
- 下拉按提供商分组;组标题点击=切提供商 + 其默认模型;组内项点击=切到该具体模型。
- 模型管理面板改为「按提供商组织」:编辑一个提供商 = 配 name/baseUrl/key/协议/默认模型 + 模型列表(自动拉取 + 手动增删 + 每模型可设窗口)。

---

## 三、当前弊端

- 现在「模型」是**扁平**的:每条配置各自带 key/baseUrl,无法表达「一个 key 对应一串模型」;切换只能按单条模型切,不是按提供商。
- **模型名手敲易错**(如 `GLM-5.1` vs `glm-5.1`),且**不感知可用列表**(得凭记忆填)。
- **不感知上下文窗口**(对应《功能盘点》),token 预算只能用保守缺省。
- 自动获取 / 模型发现完全缺失。

---

## 四、改进方向(分阶段)

> **2026-06-15 复核·实现状态**：**数据层与自动获取已落地**——提供商形态（含 `models[]`/`defaultModel`/`protocol`）+ 旧扁平配置幂等迁移 + `GET /models` 拉取（模型编辑弹窗已接入，有单测）。**前端部分接入**：模型编辑弹窗已按提供商组织（可配 `models`/`defaultModel`、可拉取列表），但**管理列表仍是扁平卡片**（每提供商一张卡、不展开 `models[]`、无分组）、**顶栏「提供商·模型」切换下拉未做**、**对话路径仍按单模型解析**。**Anthropic 协议仅 `protocol` 字段预留**，发送层未实现（详见专项调研）。

1. **数据模型**:确立 Provider 为主实体(含 models[]、defaultModel、protocol);会话改记 providerId + modelId。
2. **迁移**:现有 3 个模型配置 → 各升格为提供商(当前 model 作默认),随后自动拉取补全列表;旧会话的模型引用需兼容映射。
3. **自动获取**:`/models` 拉取 + 缓存 + 手动覆盖;内置 contextWindow 对照表(按模型)。
4. **前端**:顶栏「提供商 · 模型」单分组下拉;模型管理面板按提供商重组(含「刷新模型列表」按钮、每模型窗口可设)。
5. **协议口子**:provider 带 protocol(openai 默认);为 Anthropic 预留(发送层/provider 抽象对接调研文档)。
6. **验证**:迁移后旧会话仍能正常对话;新下拉切换即时生效;走「规划→改→测→ecc/codex 审查→迭代」固定 pipeline。

---

## 五、风险 / 注意

- **迁移兼容**:旧会话记的是单模型 id,要映射到 providerId + modelId(或保留旧引用做回退),不能让历史会话开不了。
- **列表可能很大/带前缀**(Qwen 返回 215 个、含 `ZHIPU/GLM-5` 这类前缀):下拉需支持**搜索/过滤**,且只展示该提供商实际可用的。
- **contextWindow 对照表需维护**(模型迭代快);手填覆盖优先于对照表。
- **自动获取是网络请求**:失败要优雅降级(缓存/手动),不阻塞主流程;不在每轮对话里拉。
- **协议差异**:openai 的 `/models` 与 anthropic 的 `/v1/models` 不同;获取与发送逻辑都按 protocol 分流(口子预留,本次只实现 openai)。
