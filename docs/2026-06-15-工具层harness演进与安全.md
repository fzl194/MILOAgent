---

## ① 功能要点

把当前散装的工具层(定义 / 风险分类 / 预处理 / 执行 / 结果整形散落 5 个文件)演进为「**解耦 harness + 每工具声明式契约**」:

- **共用 harness** 拥有一条固定、不可跳过的生命周期:**校验 → 归一化 → 授权 → 执行 → 整形**(可观测贯穿全程,非末尾阶段)。每阶段对应一条稳定性不变量(垃圾不入 / cwd 已解析 / 风险已闸 / 有界 / 如实 / 可观测)。
- **每个工具**只声明自己的定制(入参契约、权限判定、执行、输出整形、输出预算、只读/并发安全元数据),关注点 co-locate 到该工具自己的模块。
- **安全是首要约束**:工具必须安全执行。契约与提示词描述、参数约束**对齐 Claude Code 源码**,但安全机制因我们无内核沙箱而**比源码更严**(见 ②)。

## ② 实现逻辑

**契约(对齐 `src/Tool.ts`)**:工具 = name + inputSchema(Zod,单一真相源)+ maxResultSizeChars(必填,超此落盘)+ 元数据 isReadOnly / isConcurrencySafe(fail-closed 默认 false)+ 生命周期方法 validateInput(业务值校验)/ checkPermissions(权限接缝,取代原先设想的风险方法)/ call(执行)/ mapToolResultToToolResultBlockParam(输出整形)。校验分两层:harness 跑 schema 类型校验,工具跑业务值校验,失败都回可自纠错误给模型。工具提示词描述与参数约束与 Claude Code 保持一致。

> **与源码的刻意分歧(记录在案)**:① `outputSchema` —— Claude Code 中本就是 **optional**(`Tool.ts` 里 `outputSchema?: z.ZodType<unknown>`),本项目选择不定义,而非源码缺失;未来对关键工具可补。② `checkPermissions` 默认值 —— Claude Code 的 `TOOL_DEFAULTS.checkPermissions` 默认返回 `{ behavior: 'allow', updatedInput }`(**完全放行、交通用权限系统**);本项目因无内核沙箱,选择**更严**:工具必须自行声明风险、危险一律问,**不照搬源码的默认 allow**。

**生命周期**:校验(两层)→ 归一化(解析 cwd / 路径,授权前后各一次)→ 授权(调工具 checkPermissions → 喂中心 decide(),审批引擎 deny>ask>allow 首次匹配**不动**)→ 执行(统一超时 / 取消 / 预算)→ 整形(大结果**落盘 + 返回 preview + 文件路径**,非截断标志;read 超限则**抛错 + 续读提示**)。

**退出码语义**:非零退出不再一律判错;is_error 只表「被中断」,退出码按命令语义判定(grep / find / diff / test 退出 1 不算错)。

**安全双层(本项目特有,因无内核沙箱)**:
- 第一层(renderer,已有):classify → decide → 审批卡。危险操作一律问、永不自动、永不记忆。
- 第二层(主进程,defense-in-depth,**逐工具落地,非一刀切**):`read_file` 的 IPC handler 已重新校验——工作区根之外的设备文件(含 `\\.\`/`\\?\`/CONIN$)、canonicalize 解符号链接、超限一律**直接拒**,即使 renderer 被绕过(如聊天 UI 被注入后直调 IPC)read 路径也不越界。**`write_file` / `run_shell` 的主进程兜底为 P2 待办**:P2 落地前,write/shell 的唯一防线是 renderer 闸门(已加固:解释器 `-e/-c` 内联代码、shell 元字符链、高危 base 命令一律强制问、无 workspaceRoot 时 write_file 归 dangerous)。

run_shell:危险命令正则分类(全部「问」,不内置无条件放行;deny 规则剥 `VAR=val` 前缀防绕过);超时一击 SIGKILL 进程树(与现状对齐);大输出落盘;只读命令(ls/cat/grep)标并发安全元数据。

## ③ 当前弊端

- 一个工具的关注点散落在三处、无统一工厂:声明在 `definitions.ts`(纯 JSON schema)、执行逻辑在 `adapters/electron-tool-executor.ts`、风险分类在 `safety/classifier.ts`,改一处需同步多处、易漂移(典型:`loop.ts` 里 `resultTruncated` 永远 false 的静默谎言)。
- 无参数校验(到处 `as string`);read / write 无超时 / 预算,能挂起或撑爆上下文。
- 非零退出码一律 isError,grep / 测试被误判为失败。
- **主进程兜底不全**:`read_file` 的 IPC 已防御(设备文件 / canonicalize / 大小上限),但 `write_file` / `run_shell` 仍**零校验**——renderer 传什么执行什么,所谓「沙箱」只是 renderer 策略;renderer 一旦被绕过,write/shell 即可任意执行(`resolveUnder` 仅做相对路径拼接,绝对路径直通工作区外)。最大安全隐患,列为 P2 待办。
- 风险分类是中心 switch,不归工具自己;outputSchema 我们选择不定义(源码中本就 optional)。

## ④ 改进方向

分阶段、行为保持迁移(每阶段 typecheck + 单测全绿,走 §4 pipeline:实现 → 测试 → ecc 审查 → codex 独立审查 → 迭代):

- **P1**:搭 harness 骨架(defineTool 工厂 / 生命周期 / 预算 / 错误判别联合)+ read_file 做透(两层校验、cwd 解析、行号输出、超限抛错 + 续读、二进制守卫、ENOPT 建议);loop 用 feature flag 双路灰度。
- **P2**:输出策略落地(read 抛错续读、shell 落盘 + preview + 路径、write 存根)+ **主进程安全兜底**(write 工作区边界、read 设备文件 / 大小、shell cwd / 超时 / 输出上限在 IPC 层强制)。
- **P3**:run_shell 压测(退出码分离上报、危险正则搬进 checkPermissions、大输出落盘);classifier 风险来源可插拔,decide 不动。
- **P4**:write_file(预读 + 过期检查,Windows 内容比对兜底)+ 收口删 loop 老路径与特殊预处理、移除 flag、registry 聚合。

硬门:`classifier.test.ts` 全绿(风险来源迁移不改审批行为)。

**明确暂不引入**(留演进缝,不写死):tree-sitter 命令 AST(保留现有正则,作未来加固)、PreToolUse / PostToolUse hook(无 hook 系统,留 no-op 接缝)、outputSchema 再校验、并行调度器(留 isConcurrencySafe 元数据)。
