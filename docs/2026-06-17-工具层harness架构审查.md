## ② 实现逻辑现状(解耦性逐层评估)

### 地基解耦:优秀(世界级)

- **契约 schema-first 且单一真相源**:工具 = name + Zod inputSchema + maxResultSizeChars + isReadOnly/isConcurrencySafe + 生命周期方法。Zod 同时供运行时校验、线缆 JSON schema、TS 类型推导——三处一个源。`zodToJsonSchema` 还做了缓存与递归剥 `$schema`/`additionalProperties`,解决 GLM/DeepSeek 拒收。这是正确做法。
- **工厂 + fail-closed 默认**:未声明权限的工具默认 deny(刻意比 Claude Code 的默认 allow 更严,有 documented divergence)。安全默认正确。
- **生命周期不可跳过**:校验 → validateInput → checkPermissions → call → 整形,五段固定,且 runTool 承诺"永不抛错"——每条失败路径都返回可用结果。这是 harness 的核心不变量。
- **关注点 co-location + 可测**:首个迁移工具 read_file 把错误信息拆成纯 helper(无 I/O),可直接单测。教科书级可测性。

## ③ 当前弊端(按严重度)

### 1. 文档已漂移(信息债务,中)— **本次审查首要新发现**

2026-06-15 文档把以下项列为"P2 待办/未做",但代码实际已落地:

- **write_file 主进程兜底**:已与 read_file 同构(canonicalize + 设备文件拒绝 + 工作区边界 + mkdir)。文档仍称"零校验、唯一防线是 renderer"。
- **shell 大输出落盘**:超阈值落盘 + preview + 路径返回已就位。文档仍列为未做。

漂移后果:新会话按过时文档判断优先级,会重复做已完成的事。**建议文档立即同步。**

### 2. 双轨中间态未设收口线(架构债,高)— 唯一实质耦合

loop 当前按"工具是否在 registry"分两路:harness 路径(read_file)有完整生命周期;legacy executor(write_file/run_shell)没有。副作用:

- **schema 单一真相源被违反**:read_file 在 registry 的 Zod schema 与 definitions.ts 的手写 JSON schema 两处并存。
- 一半工具享受契约保护,另一半裸奔。
- 文档 P4 自己写了"收口删 loop 老路径"——作者知道,但未给截止线。

这是渐进迁移的必经态,本身不可耻;**可耻的是让它无限期停留。** 需要一个明确的"双轨清零"里程碑。

### 3. shell 输出编码未归一化(功能 + 次级安全,中)

主进程 shell 捕获仍是逐块 `toString()`(默认 UTF-8):Windows 中文 cmd 内置命令按 GBK 输出 → 乱码,流向模型/聊天/trace。文档 §② 已给出正确的端态设计(主进程返回原始字节,解码归 harness 整形)与过渡方案(StringDecoder + chcp 65001),设计清晰,仅未动工。

### 4. 退出码语义未分离(功能 + 安全副作用,中)

非零退出码一律判 isError。grep/find/test/diff 退出 1 被误判失败 → 模型可能误重试一个本已成功的命令,引发意外副作用。

### 5. shell 命令级安全存在架构张力(诚实表述,不可假装能补到 read/write 强度)

read/write 的路径安全已工业级双层完整。但 run_shell 在主进程层**完全没有命令级校验**(renderer 传什么执行什么)。

**关键判断(架构师立场)**:这不是"漏做",而是**无内核沙箱下的真实约束**。命令的攻击面是命令本身,主进程无法可靠审查命令(正则分类已在 renderer 做,搬进主进程也未必更安全)。因此:

- run_shell 的安全形态应与 read/write **不同**:renderer 闸门是命令安全的**唯一可信防线**;主进程只做**执行卫生**(超时 / kill tree / cwd 校验 / 落盘 / 退出码语义),不承诺命令审查。
- 文档 §6.4"逐工具落地"应按此解读:**shell 的兜底形态 ≠ read/write 的兜底形态**。强求一致是错误方向。

## ④ 改进方向(给三角色排优先级)

按"价值 × 解债 × 风险"排:

| 优先级 | 工作项 | 解什么 | 前置 |
|---|---|---|---|
| **P0·第一卡(推荐)** | **收口双轨**:write_file / run_shell 迁上 harness(进 registry、声明 Zod schema + checkPermissions + 只读/并发元数据),删 definitions.ts 重复 schema,删 loop legacy 分支 | 消除唯一实质架构债;让 write/shell 获得完整生命周期;**为退出码分离 / 危险正则搬家 / 编码元数据提供落地载体** | 无,是 unlock 后续的关键路径 |
| P0·并行小卡 | 文档同步:把 doc 2026-06-15 的过时项改正(write 兜底已做、shell 落盘已做) | 消信息债 | 无,可与第一卡并行 |
| P1·第三卡 | 退出码语义分离 + 编码归一化元数据(挂在 harness 内) | 功能 + 次级安全 | 依赖第一卡 |
| P2 | run_shell "执行卫生"补齐:退出码语义、cwd/超时/落盘在主进程固化;命令安全明确写进 design 为 renderer 闸门职责 | 诚实表述安全边界 | 依赖第一卡 |

### 收口纪律(架构师强制要求)

- **双轨必须有截止线**:第一卡完成即双轨清零,不允许新工具再走 legacy。
- **新工具一律走 defineTool**:自第一卡起,这是唯一入口。
- **安全以本双层为准,不照搬 Claude Code 的默认 allow**:已立,保持。
